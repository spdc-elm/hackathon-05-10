# 需求分析 — LLM Wiki Approach

## 问题重述

7 本医学教材存在大量内容重叠，需要：
1. 解析为结构化文本
2. 提取知识点并构建知识图谱
3. 跨教材去重压缩到 30%
4. RAG 精准问答
5. 教师对话反馈

## 核心子问题分解

### 1. 知识点粒度定义

**决策**：每章抽取 8-15 个可教学核心知识点。

理由：
- 太粗（章级别）→ 无法做跨教材对齐
- 太细（句级别）→ 关系爆炸，LLM token 消耗过高
- 每章 8-15 个 ≈ 每本书 100-200 个节点 → 7 本总计 700-1400 节点，图谱可交互

### 2. 重复判定标准

**策略**：基于文件名的精确匹配 + 语义相似度

- 同名概念文件 → 自动识别为同一知识点
- LLM 抽取时通过规范化 name 字段实现初步对齐
- 后续版本：embedding 计算 vault/concepts/ 文件间相似度

### 3. 教学连贯性保障

**方法**：通过 `prerequisite` 关系类型保证

- 每个概念的 `## 关系` section 标注前置依赖
- 删除节点前检查是否有其他节点将其作为 prerequisite
- 保留关键路径上的所有节点

### 4. 压缩比计算方式

```
原始总字数 = sum(所有教材 total_chars)
整合后字数 = sum(vault/concepts/ 所有文件字数)
压缩比 = 整合后字数 / 原始总字数
```

目标：≤ 30%

### 5. 存储格式选择

**决策**：Obsidian-flavored Markdown vault

优势：
- 人类可读、可直接用 Obsidian 打开浏览
- Git 友好（纯文本 diff）
- 知识图谱从 `[[wikilinks]]` 自动派生（零额外存储）
- 兼容静态站点生成器（Quartz/Hugo）

### 6. RAG 分块策略（下一版本）

当前 MVP 暂未实现 RAG。计划方案：

**方案 A（LLM Wiki 方式）**：
- 每个 concept MD 文件就是一个 "chunk"
- 用户提问 → 搜索 concept 文件名/aliases/定义 → 将匹配的 MD 文件内容注入 prompt
- 天然带 citation：文件路径 = 来源

**方案 B（传统 RAG）**：
- 对 vault/textbooks/ 下的章节做 500-800 字分块
- Embedding → FAISS/ChromaDB
- 混合检索 (vector + BM25)

倾向方案 A，因为与架构一致且实现更简单。
